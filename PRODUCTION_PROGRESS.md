- **1.3 — Local response cache for `intent` + `time_parse`** — Added a process-local LRU+TTL cache (`OrderedDict` + `threading.Lock`, cap 1000 entries, TTL 300s) keyed on `(label, model, sha1(system_prompt + normalize_ar(to_ascii_digits(last_user))))`. Only `intent` and `time_parse` labels participate — these are short-classification calls at `temperature=0.0` so identical normalised input deterministically produces identical output. `conversation`, `routing`, `triage` are NOT cached (too much mutable state in their inputs). Files: `llm/client.py`.
- **1.4 — Prompt size guard (defensive char cap)** — Added `_cap_message_size(messages)` helper to `llm/client.py` and wired it into `call_llm` before payload assembly. Each individual message's string content is truncated from the start (keeping the tail with a leading `…` marker) if it exceeds **6,000 chars** (per-message cap), and the oldest messages are dropped until the cumulative payload is under **24,000 chars** (about 6k tokens of headroom on a 70B context). Protects against pathological inputs — clipboard dumps, copy-pasted form text, prompt-injection payloads — that could otherwise turn one turn into a $0.01+ event. Constants live in `llm/client.py` (structural defense, not a per-environment knob). System prompt is NOT counted (it's authored by us). Non-string content (the item 1.2 structured cache shape) passes through untouched. Truncation is silent — pathological inputs are rare and observability comes through the existing `prompt_hash` shift on `llm_calls` rows. Verified 8 cases: no-op on normal-size, single oversized message → 6000 chars + `…`, boundary at exactly 6000 / 6001 chars, cumulative cap (20×2000 → drops oldest to land at 12×2000=24000), empty/None list, missing content key, non-string content untouched, end-to-end via `call_llm` sends the capped payload. ## 🩹 Post-release hotfixes

- **HF1 — Guard against `content: null` from OpenRouter** (2026-05-13) — A production session crashed on turn 2 with `AttributeError: 'NoneType' object has no attribute 'strip'` at `llm/client.py:393`. Root cause: OpenRouter / upstream sporadically returns `{"choices":[{"message":{"content": null}}]}` (e.g. when the model produced no text or only a tool-call). The crash bypassed every retry / fallback path inside `call_llm`; the user saw the bilingual app-error apology from item 4.2 instead of a normal LLM fallback. Fix: replaced `resp_json["choices"][0]["message"]["content"].strip()` with a defensive-accessor chain ending in `or ""` — null/missing content collapses to empty string and the existing "empty content" branch (3 retries + `_safe_default`) takes over from there. Pre-existing bug, not introduced by Stage 1-5; surfaced because item 4.2's catch-all now logs to the `errors` table where it could be triaged. Verified 5 cases via mocked `_session.post`: `content: None` → graceful fallback (was AttributeError pre-fix), retry rows logged as `EmptyContent`, normal string content unchanged, missing `choices` key handled, empty `choices` list handled. Files: `llm/client.py`.

- **HF2 — Remove 😊 from greeting prompt templates** (2026-05-13) — Recurrence of LLM emoji garbling: Llama 3.3 70B (via OpenRouter) started replacing the 😊 in the greeting templates with hex byte-fallback tokens (e.g. `… أحمد 484a41 424f45 434a41 23424f45 233327394f43`). Confirmed via DB: the bot's `reply` field literally contained the hex strings (`chat_logs.content` row at 10:42), `status='ok'` in `llm_calls`, valid JSON came back. Encoding pipeline was proven healthy — the May 5 confirmation cards with 8 emojis (✅📋👨‍⚕️🏥📅🕐💳💵) rendered correctly through the exact same UTF-8 round-trip. The issue is the LLM's tokenizer producing malformed byte-fallback tokens when reproducing emoji from the system-prompt examples; OpenRouter sometimes routes to quantized backends that exhibit this. Fix: removed all three 😊 from `prompts/conversation.py`'s AR/EN template-example lines (`none / not started` AR + EN, `greeting_done` AR), replacing each with natural punctuation (Arabic `،` for AR, `.` for EN). Instruction-context ⚠️ markers preserved (LLM doesn't reproduce those). `app.py`'s hard-coded initial-splash 😊 preserved (rendered by Streamlit directly; LLM never sees it). Confirmation-card emojis preserved (generated by code in `services/formatter.py`, not the LLM). Verified: no 😊 left in AR:/EN: template lines, replacements landed with correct punctuation, `prompts.conversation` imports + formats, `app.py` splash emojis untouched (2 still present). Files: `prompts/conversation.py`.

- **HF3 — Enforce dict shape on `json_mode=True` responses** (2026-05-13) — Third LLM-shape crash in the same session: turn 2 hit `AttributeError: 'list' object has no attribute 'get'` at `nodes/conversation.py:250` (`result.get("reply", "")`). Root cause: the upstream LLM returned a **valid JSON array** `[...]` when the prompt asked for an object `{...}`. `json.loads()` succeeded on the array, so the existing `JSONDecodeError` retry path didn't trigger — the wrong-typed result reached the node, which crashed on `.get()`. Same class as HF1 (null content) and HF2 (garbled emoji) — upstream model occasionally producing malformed output. Fix: added an `isinstance(result, dict)` check immediately after `json.loads(content)` in `llm/client.py`; non-dict results raise a synthetic `json.JSONDecodeError` that drops into the existing 3-retry → `_safe_default` path. `call_llm(json_mode=True)` is now contract-enforced to return a dict, matching its type signature and every caller's expectation. `_safe_default` shapes for all 5 labels are already dicts so the fallback path is consistent. Verified 6 cases via mocked `_session.post`: array response → 3 retries → dict safe_default (was AttributeError pre-fix), retry rows logged as `JSONDecodeError`, array-then-dict recovery succeeds, scalar string response also caught, normal dict response unchanged (regression), `result.get("reply", "")` works on whatever `call_llm` returns. Files: `llm/client.py`.
